## Summary

Phase 05.2 is a visual-only UI polish phase porting the Anthropic Chat Stream
design (bubble grouping + inter-group timestamps + matrix-mode CSS skin +
view-toggle pill + MatrixRain canvas Easter egg). The hard rule that
`/api/chat/route.ts` and all backend files remain untouched (D-A-02-AMENDED)
appears fully respected — no Phase 2 byte-identical contract violations
observed in the files reviewed.

Overall quality is high. Code is well-commented with extensive cross-references
to the plan/research docs and explicit callouts for every pitfall and design
decision (D-A-01..D-B-03, CD-02..CD-06, Pitfalls 2/3/4). The pure helpers
(`chat-format.ts`) are well-tested with edge cases (empty arrays, missing
timestamps, exact 5-min boundary). The body-class cleanup in `chat/page.tsx`
is correctly scoped to prevent leakage to `/admin` (Pitfall 2). The
`onFinish` reset logic in `ChatUI` correctly guards against the BL-18 v6
quirk where `onFinish` fires after `onError`.

Two warnings flag real but limited-scope issues:

1. **MatrixRain a11y/viewport gate is mount-only** — if the user toggles
   `prefers-reduced-motion`, resizes their window across the 768px boundary,
   or rotates a tablet from landscape to portrait, the RAF loop continues
   regardless. Not a security or correctness bug, but it weakens the CD-03
   defense.
2. **Canvas font string uses a CSS custom property** — `var(--font-matrix)`
   inside `ctx.font` is silently ignored by Canvas 2D (CSS vars aren't
   resolved in canvas font shorthand). The literal fallback `'Share Tech Mono'`
   works because the font is loaded by `layout.tsx`, but the `var()` ref is
   dead. The rendered output is still correct.

Info-level items cover ARIA tab-pattern strictness, a double-cast in
`ChatUI.tsx`, a Playwright test that asserts cleanup behavior under a hard
navigation that wouldn't require the cleanup, and a few small redundancies
worth noting for future maintenance.

No source modifications were made — review is read-only.

## Warnings

### WR-01: MatrixRain a11y + viewport gates run only on mount

**File:** `src/components/MatrixRain.tsx:89-98`
**Issue:** The `prefers-reduced-motion` matchMedia check and the
`window.innerWidth < 768` viewport check both run once inside the `useEffect`
when `visible` first becomes true. After mount, neither is re-evaluated:

- A user who enables "Reduce motion" in OS settings while the page is open
  continues to see the RAF loop.
- A user who resizes from 1200px (RAF active) down to 700px (or rotates a
  tablet from landscape to portrait) keeps the RAF loop running — the
  `resize` handler recomputes columns but does not cancel the animation.

CD-03 is documented as "defensive gates inside the useEffect" — that's
honored at mount, but a live `MediaQueryList` listener + a width threshold
re-check inside `resize()` would make the gate continuous. The
viewport-width case is the more realistic one (the design pill is fixed
top-right and users do resize windows during a demo).

**Fix:**
```tsx
useEffect(() => {
  if (!visible) return;
  const canvas = canvasRef.current;
  if (!canvas) return;

  const reducedMotionMql = window.matchMedia('(prefers-reduced-motion: reduce)');

  function shouldAnimate() {
    return !reducedMotionMql.matches && window.innerWidth >= 768;
  }

  // ...existing setup...

  function step() {
    if (!shouldAnimate()) {
      // Halt loop; can be resumed via resize/mql listener below.
      rafRef.current = 0;
      return;
    }
    // ...existing draw...
    rafRef.current = requestAnimationFrame(step);
  }

  function maybeStart() {
    if (shouldAnimate() && !rafRef.current) {
      rafRef.current = requestAnimationFrame(step);
    }
  }

  reducedMotionMql.addEventListener('change', maybeStart);
  window.addEventListener('resize', () => {
    resize();
    maybeStart();
  });

  if (shouldAnimate()) {
    rafRef.current = requestAnimationFrame(step);
  }

  return () => {
    if (rafRef.current) cancelAnimationFrame(rafRef.current);
    reducedMotionMql.removeEventListener('change', maybeStart);
    // ...remove resize listener...
  };
}, [visible]);
```

### WR-02: Canvas font string contains CSS variable that won't resolve

**File:** `src/components/MatrixRain.tsx:140`
**Issue:** `ctx.font = \`${RAIN.fontSize}px var(--font-matrix), 'Share Tech Mono', monospace\`;`
— Canvas 2D's `font` setter parses a CSS font shorthand but does NOT resolve
CSS custom properties (`var(...)`) against any element. Per the HTML spec,
`ctx.font` is parsed as if it were the value of the CSS `font` property on
a synthetic element with no inherited custom properties. So `var(--font-matrix)`
silently falls through to the next item in the family list (`'Share Tech Mono'`),
which happens to be loaded by `layout.tsx` and works correctly.

The visual output is correct, but the `var(--font-matrix)` reference is
effectively dead code in this position. It also means that if Joe ever
swaps the matrix font by re-pointing `--font-matrix` in CSS, the canvas
will not pick up the change — a future maintainer would expect a CSS-var
swap to propagate.

**Fix:**
```ts
// Drop the var() reference; rely on the literal family that next/font loads:
ctx.font = `${RAIN.fontSize}px 'Share Tech Mono', monospace`;
```
Or, if Joe wants single-source-of-truth for the font family, read the
resolved family from `getComputedStyle(document.body).getPropertyValue('--font-matrix')`
once at setup and interpolate the resolved value into `ctx.font`.

## Info

### IN-01: `MatrixRain` `visible={false}` branch is unreachable

**File:** `src/components/MatrixRain.tsx:209-211`, callers at `src/app/chat/page.tsx:72`
**Issue:** `chat/page.tsx` renders `{view === 'matrix' && <MatrixRain visible />}`
— the component is conditionally mounted, so when it's in the tree, `visible`
is always `true`. The `aria-hidden={!visible}` and opacity-0 fade-out branches
(`opacity-0` class, the early-return in the effect when `!visible`) are
dead. If the original intent was to allow fade-out before unmount, the
parent must mount it permanently and pass `visible` reactively.
**Fix:** Either drop the `visible` prop entirely (simpler), or move the
conditional mount up to opacity-keyed render and let the prop drive the
transition. Document whichever is chosen so a future reader understands
why the `false` branch exists or is removed.

### IN-02: Double-cast on `m.parts` in ChatUI bypasses type safety

**File:** `src/components/ChatUI.tsx:226`
**Issue:** `parts={m.parts as unknown as AssistantProps['parts']}` is a
deliberate double-cast through `unknown`, which is the TS-blessed way to
say "I know better than the inferred type." The accompanying comment
acknowledges the looseness. If AI SDK v6 ever expands `UIMessage.parts`
with a new part shape (e.g., `'file'`, `'source-document'`) that doesn't
match the narrower `TextPart | ToolPart` union, runtime code in
`MessageBubble` will silently treat it as a tool part (via
`type.startsWith('tool-')`) or be skipped, with no compile-time signal.
**Fix:** Add a runtime filter that explicitly narrows to known part types
before passing, e.g.:
```ts
const safeParts = m.parts.filter(
  (p): p is TextPart | ToolPart =>
    p.type === 'text' || p.type.startsWith('tool-'),
);
items.push(<MessageBubble key={m.id} role="assistant" position={position} parts={safeParts} />);
```
Or define a union type alias and check that AI SDK's union is a subset at a
single boundary helper (compile-time enforcement). Low priority — current
behavior is correct for today's part set.

### IN-03: Navigation-cleanup test passes for the wrong reason

**File:** `tests/e2e/cat-06-view-toggle.spec.ts:118-135`
**Issue:** The test "navigating away from /chat in matrix mode strips
body.matrix-mode (Pitfall 2)" uses `page.goto('/')`, which is a hard
navigation. On hard navigation the document is replaced entirely, so the
new `/` document starts with a fresh `<body>` and the assertion
`stillMatrix === false` would pass even if `chat/page.tsx`'s cleanup
function had been removed. The actual leak scenario Pitfall 2 protects
against is SPA-style route changes (e.g., `router.push('/admin')`), which
do NOT replace the document.
**Fix:** Use a Next.js client-side route transition to exercise the
cleanup contract. Either:
- Mount a link in `/chat` (a `<Link href="/admin/...">` or `router.push`)
  and click it, then assert.
- Use `page.evaluate(() => window.history.pushState({}, '', '/admin'))` and
  manually trigger React's effect-cleanup path — though this won't actually
  unmount the React tree without a real Next.js navigation.

Cleanest is to add a test-only nav button somewhere, or write a vitest
unit test that mounts `<ChatPage>`, sets matrix mode, unmounts, and asserts
`document.body.classList.contains('matrix-mode') === false`.

### IN-04: ViewToggle uses tablist/tab roles without tabpanel association

**File:** `src/components/ViewToggle.tsx:30-66`
**Issue:** The component declares `role="tablist"` with two `role="tab"`
buttons but does not provide `aria-controls` references to a `role="tabpanel"`
element. Strict a11y audits (axe, WAVE) flag this as an incomplete tab
pattern. The semantics are also a slight mismatch: clicking the pill
doesn't switch between two view panels in the ARIA sense — it toggles a
body class that re-skins the same content.
**Fix:** Either:
- Drop the tab roles and use a pair of toggle buttons with
  `aria-pressed="true|false"` instead of `aria-selected`. This is the
  honest semantic (toggle, not tab switch) and passes axe.
- Add `aria-controls="chat-main"` on each button (the chat panel does
  have `data-testid="chat-main"`; assign it `id="chat-main"` and
  `role="tabpanel"`). More work, less correct.

The toggle-button + `aria-pressed` route is recommended.

### IN-05: `pickGlyph()` may return undefined under strict index access

**File:** `src/components/MatrixRain.tsx:59-61`
**Issue:** `MATRIX_GLYPHS[Math.floor(Math.random() * MATRIX_GLYPHS.length)]`
declares a return type of `string` but, under `noUncheckedIndexedAccess`,
TS would type the indexed access as `string | undefined`. The project's
`tsconfig.json` does NOT enable that flag today, so this compiles. There
is also belt-and-suspenders fallback at line 164 (`col.glyphs[k] ?? pickGlyph()`)
that handles undefined defensively. If the project ever turns on
`noUncheckedIndexedAccess`, line 60 becomes a TS error.
**Fix (preemptive):**
```ts
function pickGlyph(): string {
  return MATRIX_GLYPHS[Math.floor(Math.random() * MATRIX_GLYPHS.length)] ?? 'ｱ';
}
```

### IN-06: Redundant `if (!canvas) return` inside resize closure

**File:** `src/components/MatrixRain.tsx:110-127`
**Issue:** The outer effect already does `if (!canvas) return;` before
defining `resize`, and `canvas` is captured from the closure. The
re-check inside `resize` is dead code (canvas can never be null in that
scope unless the ref is re-assigned, which it isn't). Likewise the
`if (ctx)` guards at lines 118-121 are dead — `ctx` was already
null-checked at line 101 before any `resize` call.
**Fix:** Drop the redundant guards for clarity:
```ts
function resize() {
  const w = window.innerWidth;
  const h = window.innerHeight;
  canvas.width = Math.floor(w * dpr);
  canvas.height = Math.floor(h * dpr);
  canvas.style.width = `${w}px`;
  canvas.style.height = `${h}px`;
  ctx.setTransform(1, 0, 0, 1, 0, 0);
  ctx.scale(dpr, dpr);
  // ...
}
```
Cosmetic — current code is correct but has noise.
