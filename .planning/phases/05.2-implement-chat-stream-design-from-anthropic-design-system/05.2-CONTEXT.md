# Phase 05.2: Implement Chat Stream design from Anthropic design system - Context

**Gathered:** 2026-05-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Close the visual gaps between the Anthropic Chat Stream design bundle (delivered 2026-05-01, partially ported into the codebase on 2026-04-30) and the live recruiter-facing chat surface, BEFORE Plan 05-12 LAUNCH-05. Visual-only — no changes to streaming/tool/six-gate logic, prompt caching, email gate, or PlainHtmlFallback.

**The discovery that shapes this phase:** roughly 60-70% of the bundle was already ported in an earlier pass (color tokens, light/dark CSS variables, Inter font, JD avatar header, backdrop-blurred composer, bubble-pop + typing-blink animations, custom scrollbar, basic blue/dark-grey bubbles). What remains are the bundle features that were skipped or only partially completed: bubble grouping with squared inside corners, inter-group timestamps, a working dark-mode toggle (tokens currently dead), and the matrix-mode easter-egg view (entire feature absent).

**What this phase delivers:**
1. Bubble grouping with iMessage-style tail-corner radius (squared inside corners on consecutive same-sender bubbles)
2. Inter-group timestamps shown on first message + after >5-minute gaps
3. Top-right toggle pill on `/chat` only, labeled "Light Mode" / "Dark Mode" — clicking "Dark Mode" enters matrix mode (per bundle's cheeky labeling)
4. Matrix-mode view (canvas digital-rain + green-monospace terminal restyle of chat) wired as a real toggleable view; live conversation continues to stream
5. Re-baselined cat6 Playwright e2e suite so all data-testids survive and `npm run eval` cat6 passes locally before phase close

**What this phase does NOT change:**
- `useChat` wiring, AI-SDK v6 transport, six-gate API order, prompt caching, classifier
- Email gate, FramingCopy, DisclaimerBanner, StatusBanner, PlainHtmlFallback (`/` framing page is out of scope)
- MetricCard rendering, TracePanel chevron behavior (chat variant), in-flight tool chip
- The Phase 03 fallback redirect on 2 consecutive `/api/chat` 500s
- Existing `data-testid` contracts on `msg-user`, `msg-assistant`, `chat-input`, `chat-send`, `chat-error`, `starter-prompts`, `starter-*`, `tool-progress-*`, `trace-*`, `thinking-indicator`

<decisions>
## Implementation Decisions

### Scope Items (locked)

- **D-A-01:** Bubble grouping IN. Consecutive same-sender bubbles get the bundle's tail-corner pattern: 20px on outer corners, 6px on the inside corner where the run continues (per `chat.jsx:groupMessages` + `Bubble.position` logic). Single-bubble runs ("only") get all 20px corners. Affects `MessageBubble.tsx`; needs a grouping-aware parent that walks the message list before rendering.
- **D-A-02:** Inter-group timestamps IN. Show first-message timestamp at session start; subsequent timestamps appear when the gap between consecutive messages exceeds 5 minutes (iMessage rule). No timestamp on individual bubbles — only on the timestamp dividers between groups.
- **D-A-02-AMENDED (2026-05-11, post-RESEARCH/UI-SPEC):** Timestamp source is **fully client-side**. User-message timestamp captured in client React state at `sendMessage()` call time. Assistant-message timestamp captured in client React state when `useChat` `status` flips to `'streaming'` (first chunk received). NOT persisted; NOT sent through `/api/chat`. **OVERRIDES** UI-SPEC.md "Pattern 1" (server-side `messageMetadata` callback to `toUIMessageStreamResponse`) — `src/app/api/chat/route.ts` MUST NOT be modified. Rationale: strict preservation of Phase 02 D-G-01..05 byte-identical `/api/chat` contract. Trade-off accepted: timestamps reflect client clock, not server-authoritative time. For UI-only display in transient session, this is fine. If server-authoritative time is ever needed (admin transcript ordering), Phase 04's persisted `messages.created_at` column already has it.
- **D-A-03:** Dark mode toggle IN, but as the matrix toggle (per D-B-01). Existing `:root` dark CSS tokens in `globals.css:118-162` REMAIN as dead code — no functional cost, preserves option for a true light/dark split in v2 — but the toggle pill does NOT switch them on. Real "true dark mode" is deferred (see Deferred Ideas).
- **D-A-04:** Matrix-mode easter egg IN. Bundle's two-state view toggle is implemented as-shipped: `view = 'chat' | 'matrix'`. Matrix mode renders the canvas digital-rain background (`matrix-rain.jsx` pattern, ported to TS/React) plus the terminal-restyled chat panel (green monospace, glowing borders, solid-green outgoing, outlined-green incoming). The live conversation continues to stream — matrix is a SKIN, not a scripted demo.
- **D-A-05:** Composer `+` plus button + mic button OUT. Bundle includes them as decorative affordances; we do not support attach or dictate. Composer stays at: rounded input shell + send-arrow button (current pattern).

### Toggle Behavior

- **D-B-01:** Toggle location: top-right fixed pill, `/chat` route only. Framing page `/` and admin pages have no toggle. Recruiter encounters it after passing the email gate.
- **D-B-02:** Toggle labels: "Light Mode" / "Dark Mode" (matches bundle, deliberate cheeky double-meaning where "Dark Mode" launches the matrix view). The payoff lands when the recruiter clicks. Joe selected this with awareness of the gimmick risk; this is the design-as-portfolio-statement intent.
- **D-B-03:** Persistence: SESSION-ONLY. No `localStorage`, no `prefers-color-scheme` integration, no cookies. Toggle state lives in React state on the `/chat` page; resets on page reload or new session. Rationale: portfolio-not-product framing, plus avoids the cookie-banner conversation Joe deliberately excluded from PROJECT.md.

### Scope Boundaries

- **D-C-01:** Framing page `/` is OUT of scope. No structural or token changes to `src/app/page.tsx`, `FramingCopy`, `EmailGate`, `DisclaimerBanner`, `StatusBanner`. The framing page already inherits the design tokens via the rounded card on `src/app/page.tsx:45` — no further work needed.
- **D-C-02:** Admin pages are OUT of scope. The `/admin` surface stays as it is; matrix mode does not appear there. Admin's `alwaysExpanded` TracePanel variant is not affected.
- **D-C-03:** Cat6 Playwright e2e selectors must SURVIVE — every existing `data-testid` listed in the domain section above stays unchanged. New design elements (timestamp dividers, toggle pill, matrix canvas, terminal-restyled bubble variants) get NEW selectors. The phase's UI work re-baselines `tests/e2e/chat-happy-path.spec.ts` and any other selector-coupled cat6 cases so `npm run eval` cat6 passes locally before close.
- **D-C-04:** Plan 05-12 LAUNCH-05 is unblocked by this phase. No new gates, no new requirements added to LAUNCH. The production-URL eval re-baselining stays in 05-12.

### Claude's Discretion

- **CD-01:** Bundle's `tweaks-panel.jsx` (debug/admin overlay with "Replay conversation" + "Dark mode" controls) is SKIPPED. It is design-tool chrome, not part of the production surface.
- **CD-02:** Matrix-rain canvas implementation: lazy-load only when `view === 'matrix'`. Don't ship the canvas/animation cost on default load. Use dynamic `import()` or render-gated mount.
- **CD-03:** Animation performance — match the bundle's 60fps target on desktop; downgrade or stop animation when `prefers-reduced-motion: reduce` is set, or on viewports below 768px (where canvas overhead matters more). This is a defensive default, not a hard requirement.
- **CD-04:** Terminal-restyled chat in matrix mode preserves all existing functionality: streaming, tool calls, trace panels, MetricCard, in-flight chips, error states. Visual restyle only — same React tree, different CSS classes. Don't fork the components.
- **CD-05:** Bundle's `back-btn` (left-side header button with chevron-left icon) and `replay-btn` (right-side header button with refresh icon) are SKIPPED. They are demo-app chrome (back to message list, replay scripted demo) and don't apply to a live agent.
- **CD-06:** Bundle's `chev` (downward chevron after the "Joe's Agent" name) — INCLUDE. Small detail, matches the bundle's contact-card pattern.
- **CD-07:** Composer placeholder text stays at the current copy ("Ask about my background, or edit a starter and send…"). Bundle's `iMessage` placeholder is the wrong copy for our context.

<specifics>
## Specific Ideas

- **The cheeky toggle labels** (Light Mode / Dark Mode where Dark Mode = matrix) are deliberate. Joe selected this knowing the gimmick risk and the PROJECT.md "feels gimmicky" warning. The intended payoff: the recruiter clicks "Dark Mode" expecting a normal dark theme, gets a Hollywood digital-rain Easter egg, and the design medium IS the message — "this PM thinks playfully about agentic-AI surfaces." Don't flatten this to honest "Chat / Matrix" labels under code review pressure.
- **Bundle is the final word for visuals.** The HTML's `<style>` block has every dimension, color, and animation timing. Don't reverse-engineer from screenshots; read the source.
- **Bundle has a 16-message scripted demo (`SCRIPT` in `chat.jsx`).** Do NOT port this. Our chat is real and live. Take only the rendering primitives (Bubble, TypingDots, Header, Composer, groupMessages) — leave `useAutoPlay` and the SCRIPT array.
- **Bundle's matrix `chat.jsx` includes a `Bubble.style` inline-radius override that beats CSS.** Be aware of specificity when porting — using inline radii means the matrix-mode CSS overrides for bubble corners need to be applied via `border-radius` on the parent or via `!important`, OR the radius logic moves entirely to JS and the matrix mode style sets a different radius value.

</specifics>
