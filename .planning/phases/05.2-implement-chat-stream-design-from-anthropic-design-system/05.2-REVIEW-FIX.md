## Fixed Issues

### WR-02: Canvas font string contains CSS variable that won't resolve

**Files modified:** `src/components/MatrixRain.tsx`
**Commit:** `64ad14b`
**Applied fix:** Resolve `--font-matrix` at setup via
`getComputedStyle(document.body).getPropertyValue('--font-matrix')` and use the
literal family in the canvas font shorthand. This preserves single-source-of-
truth: if Joe ever re-points `--font-matrix` in CSS, the canvas now picks up
the change (instead of silently falling through to the literal next-in-list
fallback). Computed once at effect setup (not per frame). Falls back to
`'Share Tech Mono'` when the variable is unset (defensive — the production
case is `next/font`'s generated family name, e.g. `__Share_Tech_Mono_abc123`).

### WR-01: MatrixRain a11y + viewport gates run only on mount

**Files modified:** `src/components/MatrixRain.tsx`
**Commit:** `37ec3fa`
**Applied fix:** Replaced the mount-only `return early` checks with a
`shouldAnimate()` helper that re-evaluates `prefers-reduced-motion` and the
768px viewport threshold on every frame. Added:

- `reducedMotionMql.addEventListener('change', maybeStart)` so toggling OS
  reduce-motion mid-session halts/resumes the RAF loop.
- `onResize` wrapper that calls `resize()` then `maybeStart()` so crossing
  the 768px threshold in either direction halts/resumes the loop.
- `step()` itself checks `shouldAnimate()` at the top and bails (setting
  `rafRef.current = 0`) when conditions no longer hold; `maybeStart()`
  resumes it cleanly on the next eligible event.
- Cleanup function now removes both the resize listener and the MQL change
  listener, and resets `rafRef.current = 0` so a stale RAF id is never
  cancelled twice in StrictMode.

Also updated the file-level docblock to reflect that CD-03 gates are now
continuous, not mount-only, and added inline `WR-01 fix:` comment markers at
the new `shouldAnimate()` helper and the in-step gate so future maintainers
understand the change boundary.
