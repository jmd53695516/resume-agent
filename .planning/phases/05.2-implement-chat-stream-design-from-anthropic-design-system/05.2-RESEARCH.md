# Phase 05.2: Implement Chat Stream design from Anthropic design system — Research

**Researched:** 2026-05-11
**Domain:** Visual UI close-out — Tailwind v4 / React 19 / AI SDK v6 / Next.js 16 App Router
**Confidence:** HIGH

## Summary

The phase closes the visual gap between the (already 60-70% ported) Anthropic Chat Stream design bundle and the live recruiter chat. The remaining work is four concrete, well-bounded chunks: bubble grouping with iMessage tail-corner radii, inter-group timestamps, a top-right Light/Dark pill (where "Dark" = matrix mode), and the matrix view itself (canvas digital-rain background + green-monospace terminal restyle of the existing chat tree). Everything is visual; the streaming/tool/six-gate logic is untouched.

Three implementation questions had non-obvious answers and now have evidence-backed recommendations:

1. **Timestamps:** AI SDK v6 `UIMessage` has NO native `createdAt` field [VERIFIED: `node_modules/ai/dist/index.d.ts:1659-1683`]. The canonical pattern is the optional `metadata` slot — typed via `UIMessage<MessageMetadata>`. User-message timestamps stamp client-side via `sendMessage({ text, metadata: { createdAt: Date.now() } })`; assistant-message timestamps stamp server-side via `toUIMessageStreamResponse({ messageMetadata: ({ part }) => part.type === 'start' ? { createdAt: Date.now() } : undefined })`. This is a small, additive, type-safe edit to `/api/chat/route.ts:458` — does NOT alter the six-gate flow or any existing field.

2. **Matrix lazy-mount:** `next/dynamic` with `{ ssr: false }` works inside Client Components only — `app/chat/page.tsx` already has `'use client'`. Mount `<MatrixRain visible={view === 'matrix'} />` via `dynamic(() => import('@/components/MatrixRain'), { ssr: false })` from inside ChatUI. This keeps the canvas + RAF loop out of the default chat-mode bundle.

3. **Matrix CSS scope:** Tailwind v4's `@custom-variant` cannot be applied via inline radii overrides cleanly. The right pattern is a `body.matrix-mode` block in `globals.css` (analogous to the existing `.dark` block at §118-162) plus a `useEffect`-managed body-class toggle in `app/chat/page.tsx`, with the inline-radius computation moving to JS so matrix-mode CSS can override via class-based rules — eliminating the `!important` battle the bundle hints at.

**Primary recommendation:** Treat this phase as four small, parallelizable tasks:
- T1: bubble grouping + radius-via-prop (touches `MessageBubble.tsx` + adds optional grouping wrapper in `ChatUI.tsx`)
- T2: timestamp pipeline (touches `/api/chat/route.ts` + `ChatUI.tsx` + new `<TimestampDivider>` component)
- T3: toggle pill + body-class effect (touches `app/chat/page.tsx`, new `<ViewToggle>` component, font added in `app/layout.tsx`)
- T4: matrix-mode CSS block (in `globals.css`) + `<MatrixRain>` ported component (canvas, dynamically imported)
- T5: e2e re-baseline (add view-toggle and timestamp-divider testids to existing specs; new specs for toggle round-trip)

### Pattern 1: Type-safe message metadata (timestamps)

**What:** Extend the AI SDK v6 `UIMessage` generic with a `MessageMetadata` shape carrying `createdAt: number`. Stamp on user side at `sendMessage`, on assistant side via `messageMetadata` callback.

**When to use:** Inter-group timestamps (D-A-02). Required because `UIMessage` has no native `createdAt` field [VERIFIED: `node_modules/ai/dist/index.d.ts:1659-1683` shows only `id`, `role`, `metadata?`, `parts`].

**Example — define the typed message:**

```typescript
// src/lib/chat-types.ts (NEW, ~10 LOC)
import type { UIMessage } from 'ai';

export type MessageMetadata = {
  createdAt?: number; // epoch ms; optional so historical messages without it still render
};

export type ResumeAgentUIMessage = UIMessage<MessageMetadata>;
```

**Server side — `src/app/api/chat/route.ts:458`:**

```typescript
// BEFORE:
return result.toUIMessageStreamResponse();

// AFTER:
return result.toUIMessageStreamResponse({
  messageMetadata: ({ part }) => {
    if (part.type === 'start') {
      return { createdAt: Date.now() };
    }
  },
});
```

This is a NEW return-shape addition. The streamed UI-message-chunks now include a `message-metadata` chunk type that the client parses into `message.metadata`. **It does NOT change the six-gate flow, classifier, tools, or any persisted DB shape.** Existing assistant text/tool parts are unchanged. [VERIFIED: `node_modules/ai/dist/index.d.ts:2147-2149, 2273-2276` shows `message-metadata` as an additive chunk type the client renders into `metadata?`.]

**Client side — `src/components/ChatUI.tsx`:**

```typescript
  if (!input.trim() || isStreaming) return;

```

[VERIFIED: `node_modules/ai/dist/index.d.ts:3804-3806` shows `sendMessage` accepts `metadata?: InferUIMessageMetadata<UI_MESSAGE>` on the text-form overload.]

**Format with date-fns:**


The bundle uses `'10:42 AM'` (uppercase AM/PM). date-fns `'a'` token = uppercase AM/PM; `'aaa'` = lowercase. Use `'h:mm a'`.

### Pattern 2: Bubble grouping with `position` prop

**What:** Compute `position: 'only' | 'first' | 'middle' | 'last'` for each message in a single pass over `messages`, pass it as a prop into `MessageBubble`, and let MessageBubble derive its inline `borderRadius` style from position.

**When to use:** D-A-01 — consecutive same-sender bubbles get squared inside corners.

**Example — pure helper:**

```typescript
// src/lib/chat-format.ts (NEW)
import type { ResumeAgentUIMessage } from './chat-types';

export type BubblePosition = 'only' | 'first' | 'middle' | 'last';

/**
 * Walk the message list once and emit a render plan: for each message,
 * its sender-run position. Used by ChatUI to attach `position` to MessageBubble.
 *
 * Same-sender consecutive messages form a "run". Position is:
 *   - 'only'   — single-message run
 *   - 'first'  — first of a multi-message run
 *   - 'middle' — interior of a 3+ run
 *   - 'last'   — last of a multi-message run
 *
 * The sender role is collapsed to 'me' | 'them' for grouping symmetry with
 * the bundle's logic.
 */
export function computePositions(
  messages: Pick<ResumeAgentUIMessage, 'id' | 'role'>[],
): Map<string, BubblePosition> {
  const out = new Map<string, BubblePosition>();
  for (let i = 0; i < messages.length; i++) {
    const m = messages[i];
    if (m.role === 'system') continue;
    const prev = i > 0 ? messages[i - 1] : null;
    const next = i < messages.length - 1 ? messages[i + 1] : null;
    const sameAsPrev = prev && prev.role === m.role;
    const sameAsNext = next && next.role === m.role;
    if (!sameAsPrev && !sameAsNext) out.set(m.id, 'only');
    else if (!sameAsPrev && sameAsNext) out.set(m.id, 'first');
    else if (sameAsPrev && sameAsNext) out.set(m.id, 'middle');
    else out.set(m.id, 'last');
  }
  return out;
}

/**
 * iMessage-style 5-minute rule: show a timestamp at index 0 and any time
 * the gap between message[i-1] and message[i] exceeds GAP_MS.
 */
const GAP_MS = 5 * 60 * 1000;

export function shouldShowTimestampBefore(
  messages: Array<{ metadata?: { createdAt?: number } }>,
  i: number,
): boolean {
  if (i === 0) return true;
  const prev = messages[i - 1].metadata?.createdAt;
  const cur = messages[i].metadata?.createdAt;
  if (!prev || !cur) return false; // historical message without timestamp — silent
  return cur - prev > GAP_MS;
}
```

[VERIFIED: bundle pattern in `design-bundle/project/chat.jsx:77-88` (`groupMessages`) and `chat.jsx:90-115` (`Bubble` radius math). My helper above matches the bundle's behavior but is restructured for streaming — bundle walks a static array; ours walks a live, growing one.]

**Example — `MessageBubble.tsx` accepts `position`:**

```typescript
// src/components/MessageBubble.tsx — add `position` to props union
type MessageBubbleProps =
  | { role: 'user'; text: string; position: BubblePosition; ... }
  | { role: 'assistant'; parts: Part[]; position: BubblePosition; ... };

function radiusFor(role: 'user' | 'assistant', position: BubblePosition): React.CSSProperties {
  // 'me' = user (right-aligned blue), 'them' = assistant (left-aligned grey)
  const isMe = role === 'user';
  const r: React.CSSProperties = {
    borderTopLeftRadius: 20,
    borderTopRightRadius: 20,
    borderBottomLeftRadius: 20,
    borderBottomRightRadius: 20,
  };
  if (isMe) {
    if (position === 'first' || position === 'middle') r.borderBottomRightRadius = 6;
    if (position === 'middle' || position === 'last')  r.borderTopRightRadius    = 6;
  } else {
    if (position === 'first' || position === 'middle') r.borderBottomLeftRadius  = 6;
    if (position === 'middle' || position === 'last')  r.borderTopLeftRadius     = 6;
  }
  return r;
}
```

[VERIFIED: bundle `chat.jsx:90-106` — same logic, bundle uses `from === 'me'`; we use `role === 'user'`.]

**Why a prop, not internal computation:** `MessageBubble` doesn't see the message list — it only sees its own message. Position requires neighbor inspection. Computing once in `ChatUI` (or in a memoized helper called from `ChatUI`) and passing the result down is the React-idiomatic pattern.

**Performance note:** CHAT-10 caps conversations at 30 turns. With grouping logic running on every render, that's 30 × ~3 ops per message = trivial. `useMemo` is overkill at this scale; a bare call is fine. If profiling later shows it's a hotspot, wrap in `useMemo([messages])`.

### Pattern 3: Inline-radius override resolves matrix-mode CSS specificity

**What:** Bundle (`design-bundle/project/Chat Stream.html:342-364`) overrides bubble appearance in matrix mode but NOT bubble radii — bubbles keep their JS-set radii in matrix mode. The "Specifics" note about inline-style override is about specificity battles only — since we always set radius via inline style, matrix-mode CSS for `body.matrix-mode .bubble-me` overrides only `background`/`color`/`border`/`box-shadow`, all of which the bundle respects.

**Decision:** Keep radius as inline style. Matrix mode does NOT need to override radii (bundle CSS confirms this). No `!important` needed.

### Pattern 4: Toggle pill state lives one level above ChatUI

**What:** `view: 'chat' | 'matrix'` lives in `app/chat/page.tsx`, passed down to `ChatUI`. A `useEffect` in `chat/page.tsx` toggles `document.body.classList` for global styling.

**When to use:** D-B-01 (toggle is page-level), CONTEXT integration-points note ("View-toggle state should live HERE one level up from ChatUI").

**Example — `src/app/chat/page.tsx`:**

```typescript
'use client';
import { useEffect, useState } from 'react';
import { useRouter } from 'next/navigation';
import { ChatUI } from '@/components/ChatUI';
import { ViewToggle } from '@/components/ViewToggle';

export default function ChatPage() {
  const router = useRouter();
  const [sessionId, setSessionId] = useState<string | null>(null);
  const [hydrated, setHydrated] = useState(false);
  const [view, setView] = useState<'chat' | 'matrix'>('chat');

  useEffect(() => {
    setHydrated(true);
    const id = typeof window !== 'undefined'
      ? sessionStorage.getItem('session_id')
      : null;
    if (!id) { router.replace('/'); return; }
    setSessionId(id);
  }, [router]);

  // Body-class side effect — adds .matrix-mode while on this view.
  // Cleanup runs on unmount (route change to /admin etc.) so the class
  // doesn't leak across navigations.
  useEffect(() => {
    if (view === 'matrix') {
      document.body.classList.add('matrix-mode');
      return () => document.body.classList.remove('matrix-mode');
    }
  }, [view]);

  if (!hydrated || !sessionId) return null;

  return (
    <>
      <ViewToggle view={view} onChange={setView} />
      <ChatUI sessionId={sessionId} view={view} />
    </>
  );
}
```

**Why body-class over `<html data-view>` or wrapping div:** Matrix mode needs to:
1. Cover the entire viewport with the canvas (it's `position: fixed; inset: 0`)
2. Restyle ALL chat-surface tokens (`--me`, `--them`, `--panel`, `--composer-bg`, etc.)

A wrapping `<div className="matrix-mode">` would scope tokens to that subtree but the body's gradient background (from `globals.css:171-177`) wouldn't switch. Body class is the only place where both the body-level gradient AND descendant tokens flip together with the bundle's CSS unchanged.

**Why `useEffect` cleanup is critical:** When the user navigates from `/chat` to `/admin/foo`, the chat page unmounts. Without cleanup, `body.matrix-mode` persists and the admin page paints itself green. The cleanup function in the effect strips the class on unmount.

**StrictMode double-invocation:** In dev, React 19 StrictMode runs the effect twice (mount → cleanup → mount). Net result: class added → removed → added — final state correct. Production builds run exactly once. [CITED: React 19 StrictMode docs.]

### Pattern 5: Lazy-mount canvas with `next/dynamic`

**What:** Render the matrix-rain canvas only when `view === 'matrix'`, and only download the component code when first needed.

**When to use:** CD-02 — don't ship canvas/animation cost on default load.

**Example — `src/components/ChatUI.tsx`:**

```typescript
import dynamic from 'next/dynamic';

// Lazy import — code-split, never SSR (canvas is client-only).
// `loading` is null because the matrix view fades in over 0.4s
// (per Chat Stream.html:404-411 .matrix-stage transition) — a flash
// of nothing for the first frame is invisible.
const MatrixRain = dynamic(
  () => import('./MatrixRain').then((m) => ({ default: m.MatrixRain })),
  { ssr: false, loading: () => null },
);

export function ChatUI({ sessionId, view }: ChatUIProps) {
  // ... existing useChat etc.
  return (
    <main className={...}>
      {view === 'matrix' && <MatrixRain visible />}
      {/* ... existing header / scroll region / composer */}
    </main>
  );
}
```

[VERIFIED: Next.js docs — `ssr: false` works only in Client Components in App Router. `ChatUI.tsx:1` is `'use client'` — qualifies.]

**Important nuance:** The bundle's MatrixRain takes a `visible` prop and uses it to gate the RAF loop. Because we conditionally MOUNT the component (`{view === 'matrix' && <MatrixRain />}`), the loop only exists when matrix is on. The `visible` prop becomes redundant under conditional mount — but keep it (defensive: lets us animate the fade-in without flashing).

### Pattern 6: Canvas + RAF in React 19 StrictMode

**What:** `useEffect` with `requestAnimationFrame` + `cancelAnimationFrame` cleanup. StrictMode-safe.

**Example — `src/components/MatrixRain.tsx`:**

```typescript
'use client';
import { useEffect, useRef } from 'react';

export function MatrixRain({ visible }: { visible: boolean }) {
  const canvasRef = useRef<HTMLCanvasElement>(null);
  const rafRef = useRef<number>(0);

  useEffect(() => {
    if (!visible) return;
    const canvas = canvasRef.current;
    if (!canvas) return;

    // Respect prefers-reduced-motion (CD-03)
    const reduce = window.matchMedia('(prefers-reduced-motion: reduce)').matches;
    if (reduce) return; // canvas stays mounted but no animation

    // Viewport gate (CD-03): skip animation on small viewports
    if (window.innerWidth < 768) return;

    const ctx = canvas.getContext('2d');
    if (!ctx) return;

    // ... RAF loop (port from design-bundle/project/matrix-rain.jsx:88-171)
    const step = () => {
      // ... draw frame
      rafRef.current = requestAnimationFrame(step);
    };
    rafRef.current = requestAnimationFrame(step);

    return () => {
      cancelAnimationFrame(rafRef.current);
    };
  }, [visible]);

  // ... resize effect (separate useEffect)
  // ... return JSX (canvas + vignette + scanlines, see bundle line 173-179)
}
```

**StrictMode safety:** Each effect mount allocates one RAF id and stores it in `rafRef.current`. On unmount cleanup, `cancelAnimationFrame(rafRef.current)` cancels the most recent loop. The double-invoke produces: mount → store id A → cleanup cancels A → mount → store id B → (eventual real cleanup cancels B). One loop runs at any time. ✅

**Cited risk:** [GitHub issue facebook/react#36284 noted some StrictMode + concurrent edge cases where cleanup may not fire reliably.] Mitigation: in dev, watch for "two animation loops" by logging from inside `step()` if needed. Production builds skip StrictMode double-invoke entirely.

### Anti-Patterns to Avoid

- **Don't add `localStorage` for view persistence.** D-B-03 explicitly excludes it. Adding it would also drag in the cookie-banner conversation that PROJECT.md deliberately excluded.
- **Don't introduce a `tailwind.config.js`.** CLAUDE.md mandates v4 CSS-first. New tokens go in `@theme inline { ... }` in `globals.css`.
- **Don't fork ChatUI / MessageBubble for matrix mode.** CD-04 requires the SAME React tree with different CSS. Forking would double maintenance and break feature-parity (TracePanel, MetricCard, in-flight chip all need to render in matrix mode unchanged in structure).
- **Don't bypass the `data-testid` contract.** Every existing testid stays unchanged (D-C-03). New testids are PURELY ADDITIVE.
- **Don't move the `submit()` logic out of ChatUI.** It's coupled to `useChat`'s `sendMessage`. Adding `metadata: { createdAt: Date.now() }` to the existing call is the entire change.
- **Don't memoize `computePositions` aggressively.** With CHAT-10 capping at 30 turns, the function runs in microseconds. `useMemo` adds noise without measurable benefit.
- **Don't auto-scroll bubble rendering for matrix mode separately.** ChatUI's `bottomRef.current?.scrollIntoView({ behavior: 'smooth' })` works for both views — same DOM, same scrollIntoView contract.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| 12-hour timestamp formatting | Custom `formatTime(epoch): string` | `date-fns@4` `format(new Date(epoch), 'h:mm a')` | Locale handling, AM/PM casing, single vs double-digit hour all subtle. date-fns is already the project standard per stack docs. |
| Message timestamp transport | Custom side fetch / Supabase select / data part hack | AI SDK v6 `messageMetadata` server callback + `sendMessage({ metadata })` client | First-party SDK feature; type-safe; serializes through the existing SSE stream; zero new endpoints. |
| Bubble grouping | Component that imperatively walks the DOM | Pure helper `computePositions(messages)` returning `Map<id, position>` | Render-time pure function is testable, deterministic, predictable. DOM walking fights React's render model. |
| Lazy code-splitting the canvas | Manual `import()` + Suspense boundary | `next/dynamic({ ssr: false })` | Built into Next.js; handles SSR exclusion correctly; no Suspense boundary needed for a fixed-position bg layer. |
| `prefers-reduced-motion` detection | Custom hook + global state | `window.matchMedia('(prefers-reduced-motion: reduce)').matches` inside the existing canvas effect | One-line check at effect start; no need for re-render reactivity (a recruiter who toggles their OS setting mid-chat is not a real path). |
| Body-class lifecycle | Pass class through layout, render-prop pattern | `useEffect` in `chat/page.tsx` with cleanup | Standard React idiom; cleanup handles route changes; one place to look for the side effect. |

**Key insight:** The AI SDK v6 metadata pipeline is the single biggest "don't hand-roll" win here. Without it, timestamps would require a side endpoint (`GET /api/messages?session=X`), polling (or cache invalidation), and divergence from the in-memory `messages` array — for ~6 LOC of additional metadata wiring you get every UIMessage tagged at the moment of stream-start.

## Common Pitfalls

### Pitfall 1: Timestamp on user messages without `messageMetadata` callback

**What goes wrong:** Adding `messageMetadata` server-side stamps assistant messages but not user messages. User messages are created on the client via `sendMessage` before the server sees them.

**Why it happens:** The server's `messageMetadata` callback fires on stream events (`start`, `finish`) for the assistant response only. User messages exist in `useChat`'s state already with whatever the client passed.

**How to avoid:** Stamp user messages on the client at `sendMessage` time. The `metadata` parameter on `sendMessage({ text, metadata })` writes directly into the user message's `metadata` field. Both sides stamp at "creation moment" — symmetric.

**Warning signs:** During implementation, log `messages.map(m => ({ role: m.role, ts: m.metadata?.createdAt }))` — every message should have a `ts`. If user messages have `undefined`, the client `metadata` arg is missing.

### Pitfall 2: Body-class leak on route change

**What goes wrong:** User toggles to matrix mode on `/chat`, then navigates to `/admin/sessions`. The admin page is now painted green-on-black because `body.matrix-mode` is still set.

**Why it happens:** A `useEffect` that adds a class without a cleanup function persists the class across the React tree's lifetime — but `<body>` lives outside the React tree.

**How to avoid:** Always return a cleanup function from the `useEffect` that adds the class:
```typescript
useEffect(() => {
  if (view === 'matrix') {
    document.body.classList.add('matrix-mode');
    return () => document.body.classList.remove('matrix-mode');
  }
}, [view]);
```

**Warning signs:** Manual smoke: toggle matrix → navigate to `/admin` → if admin is green, cleanup is missing.

### Pitfall 3: Hydration mismatch on `view` initial state

**What goes wrong:** Server SSRs the page assuming `view='chat'`; client hydrates and reads from somewhere (URL? sessionStorage?) that gives `view='matrix'`. React 19 throws hydration mismatch.

**Why it happens:** Persisted state + SSR + initial-render divergence. In our case D-B-03 says NO persistence — initial state is always `'chat'` — so this is a non-issue. But if anyone later "improves" by adding URL persistence, this trap reappears.

**How to avoid:** Keep `view` initial state as a literal `useState<'chat' | 'matrix'>('chat')`. No `() => sessionStorage.getItem(...)` initializer. The `chat/page.tsx` already uses a `hydrated` flag that gates render — same pattern.

**Warning signs:** React error in console: `Hydration failed because the server rendered HTML didn't match the client.`

### Pitfall 4: `next/dynamic({ ssr: false })` in a Server Component

**What goes wrong:** Dynamic import the matrix component from a Server Component → Next.js throws `'ssr: false' is not allowed in Server Components`.

**Why it happens:** Next 16 enforces this at build time.

**How to avoid:** Place the `dynamic()` call inside a Client Component. Both `app/chat/page.tsx` and `components/ChatUI.tsx` are `'use client'` — either is fine. Put it in `ChatUI.tsx` so it's co-located with the conditional render.

**Warning signs:** Build fails with `× ssr: false is not allowed with next/dynamic in Server Components`.

### Pitfall 5: `vitest` passes, `next build` fails

**What goes wrong:** Memory feedback `feedback_local_vs_vercel_build.md` — vitest tolerates type loosenesses that `next build` rejects (especially "implicit any" in JSX, type-only imports in client/server boundary, etc.).

**Why it happens:** vitest uses a different TS transformer than Next's webpack/Turbopack pipeline.

**How to avoid:** Run BOTH after every TypeScript change:
```bash
npx tsc --noEmit && npm run build
```
Both must succeed before declaring TS work done.

**Warning signs:** Local `npm run test` green; Vercel deploy red. By then it's already broken in prod-preview pipeline.

### Pitfall 6: 2-error fallback redirect breaks `view` state

**What goes wrong:** `ChatUI.tsx:38-45` redirects to `/?fallback=1` after 2 consecutive `/api/chat` 500s. If the user is in matrix mode when this fires, the redirect lands them on the framing page where:
- The body still has `class="matrix-mode"` (cleanup hasn't fired yet — the navigation is via `router.push`, which DOES unmount the page → effect cleanup DOES fire).

Actually traceable: `router.push('/?fallback=1')` unmounts `chat/page.tsx`, the cleanup function strips `body.matrix-mode`, framing page renders normally. ✅ No bug — but worth verifying with manual smoke during phase close-out.

**Why it happens:** Effect cleanup is guaranteed on unmount in React 19.

**How to avoid:** Trust the cleanup. Verify manually by inducing the redirect in dev (set the error counter to 2 via a debugger and trigger a third 500).

**Warning signs:** Framing page shows green Matrix styling — only possible if the effect cleanup didn't run, which would be an effect-cleanup React bug.

### Pitfall 7: Auto-scroll bottom calculation in matrix mode

**What goes wrong:** ChatUI scrolls to `bottomRef` on every new message (`ChatUI.tsx:63-65`). Matrix mode's bubbles have different padding/font-size/line-height than chat mode. If layout reflow happens after `scrollIntoView` is called, the user lands above the latest message.

**Why it happens:** `scrollIntoView({ behavior: 'smooth' })` runs against the layout AT THE MOMENT OF CALL. If a font swap (Inter → Share Tech Mono) reflows mid-scroll, the destination changes.

**How to avoid:**
1. Verify `next/font/google` for Share_Tech_Mono uses the same `display: swap` pattern as Inter at `layout.tsx:5-9` — preventing FOUT-style reflow flicker.
2. If reflow is observed, change scroll trigger from `[messages, status]` (current) to `[messages, status, view]` so a view-switch also re-scrolls.

**Warning signs:** Manual smoke: send a few messages in chat mode, toggle to matrix, observe whether the scroll position lands at the latest message.

### Pitfall 8: AI SDK v6 chunk validation rejects unknown metadata shape

**What goes wrong:** If `useChat` is initialized with a typed `<ResumeAgentUIMessage>` AND a `messageMetadataSchema` Zod validator (an opt-in feature), an unrecognized field would throw.

**Why it happens:** Defensive validation. We don't enable it.

**How to avoid:** Don't pass `messageMetadataSchema` to `useChat`. The TypeScript generic `<ResumeAgentUIMessage>` gives compile-time safety; runtime validation is opt-in and not needed for this single-field metadata.

**Warning signs:** `Cannot parse message metadata` errors in console — only triggers if someone adds the schema option.
