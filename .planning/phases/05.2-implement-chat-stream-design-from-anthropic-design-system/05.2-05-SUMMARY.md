# Phase 05.2 Plan 05: MatrixRain canvas (lazy-mounted, conditional, defensive) Summary

**`src/components/MatrixRain.tsx` (237 LOC) ports the design-bundle's matrix-rain.jsx to TypeScript + React 19 with all bundle tunables verbatim (fontSize=18, trailFade=0.06, flipChance=0.04, speedMin=0.35, speedMax=0.95, headWhiteChance=0.72), the exact glyph palette (half-width katakana + digits + Latin uppercase + symbols), and two defensive CD-03 gates (prefers-reduced-motion early-return + viewport <768px early-return). `src/app/chat/page.tsx` lazy-imports it via `next/dynamic({ ssr: false, loading: () => null })` with named-export indirection and conditionally mounts it only when `view === 'matrix'` — zero canvas/RAF cost in default chat mode (CD-02 enforced structurally).   - `pickGlyph()` + `makeColumn(rowsTall)` pure helpers extracted to module scope
  - `Column` TS type (`{ y, speed, length, glyphs }`) replaces the bundle's untyped object literal
  - `MatrixRain({ visible })` accepts a single `visible: boolean` prop
  - `useRef<HTMLCanvasElement>(null)` for canvas ref + `useRef<number>(0)` for RAF id (StrictMode-safe per RESEARCH §Pattern 6 + Assumption A5)

  - `resize()` paints `canvas.width = floor(w * dpr)` + `canvas.style.width = ${w}px` for crisp HiDPI rendering; rebuilds `cols` array on resize; registered/removed on window resize event
  - `step()` RAF loop: per-frame translucent black fillRect produces decay-trail; for each column it (a) re-randomizes trail glyphs at flipChance probability, (b) draws trail back-to-front with green gradient fade by t-position, (c) paints a fresh head glyph (72% white-tinted, 28% pure green) with green shadowBlur 8 glow, (d) unshifts head onto trail buffer, (e) advances `col.y += col.speed`, (f) probabilistically resets the column when it falls past the viewport bottom
  - JSX returns three layers inside a `fixed inset-0 z-[1] pointer-events-none` stage div: the canvas itself, a radial-gradient vignette overlay (`mix-blend-multiply`), and a repeating-linear-gradient scanlines overlay (`mix-blend-multiply opacity-60`)
  - `aria-hidden={!visible}` so screen readers ignore the decorative canvas
  - `transition-opacity duration-[400ms]` matches the bundle's 0.4s fade-in (UI-SPEC line 388)
  - Cleanup function in useEffect cancels RAF + removes resize listener — StrictMode-safe (mount → cancel → mount → cancel ends with one loop alive)

## Decisions Made

- **Plan template followed verbatim**: The plan supplied a near-complete component skeleton (~160 LOC of plan-action code block + comments). Port was a faithful copy. The 237 LOC final file = the 160 LOC skeleton + the same RAF loop body + the JSDoc header (which the plan's skeleton already included as commentary). Zero structural deviation; zero tunable changes.
- **Named-export indirection in `dynamic()`**: `.then((m) => ({ default: m.MatrixRain }))` is required because MatrixRain is named-exported. Could have been worked around with `export { MatrixRain as default }` but the plan explicitly asks for `export function MatrixRain` and the indirection is a documented Next.js pattern.
- **Conditional mount `view === 'matrix' && <MatrixRain visible />`** (not always-mount-pass-visible-prop): RAF loop should ONLY exist when matrix is active. Always-mounting with a visible prop would still allocate the canvas DOM + run the visible-prop check every render. The conditional pattern is structurally cheaper AND clearer in code review.
- **`loading: () => null`** rather than a spinner or fallback: the matrix-stage's own `transition-opacity duration-[400ms]` covers the brief load. A spinner would be jarring; a null fallback paints nothing for ~50ms while the chunk downloads, then the stage fades in.
- **MatrixRain mounted FIRST in the Fragment** (before ViewToggle + ChatUI): DOM order = visual stack order (z-index 1 bg → z-index 5 panel → z-index 50 toggle). Combined with `position: fixed inset-0 pointer-events-none` the canvas covers the viewport without blocking clicks.
- **ChatUI prop surface UNCHANGED**: Plan 04's SUMMARY anticipated this plan adding a `view` prop. In practice the body-class CSS cascade (Plan 02) handles everything view-aware EXCEPT the canvas mount, and the canvas mount lives at page level (sibling of ChatUI). No ChatUI edit needed — keeps the diff surgical and avoids re-rendering ChatUI on view changes.
