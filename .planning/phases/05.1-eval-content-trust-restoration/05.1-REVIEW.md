## Warnings

### WR-01: Deflection-skip path missing from cat4-judge and cat5

**File:** `src/lib/eval/cats/cat4-judge.ts:90-99`, `src/lib/eval/cats/cat5.ts:79-83`
**Issue:** Phase 05.1 Item #7's stated motivation (`05.1-CONTEXT.md` D-E-02 and the cat1-fab-014 reproduction in runId `vstFDlWpoKcyGH29w2KKs`) is that `ipLimiter10m` trips mid-run and the canned `DEFLECTIONS.ratelimit` text gets fed to the judge as a real model response — producing false fabrication / persona-fail rationales. This phase fixed cat1 (15 cases, biggest blast radius) and cat3 (6 cases). But cat4-judge and cat5 use the same `callAgent({ ... })` helper, destructure only `{ response }`, and discard `deflection`. The same mid-run rate-limit / spend-cap / turn-cap / borderline / classifier deflection will still be silently graded by `judgeVoiceFidelity` (cat4-judge) and `judgePersona` (cat5).

This is annotated as deferred in deferred-items.md per Joe's scope-set, but the failure mode is identical. Worth flagging because:
1. `cat5.ts:107` includes a `judge.verdict.score >= 4` warmth gate that will produce noisy 1-3 scores against the deflection vocabulary ("I'm taking a breather for the day").
2. `cat4-judge.ts` has a 4.0 aggregate gate; one deflection-graded-as-voice-sample drags the aggregate down.

**Fix:** Apply the same skip pattern from `cat1.ts:73-92` and `cat3.ts:47-66`:
```ts
const { response, deflection } = await callAgent({ targetUrl, prompt: c.prompt, sessionId });
if (deflection !== null) {
  const result: EvalCaseResult = {
    case_id: c.case_id,
    category: 'cat4-judge', // or 'cat5'
    prompt: c.prompt,
    response,
    judge_score: null,
    judge_verdict: null,
    judge_rationale: `skipped: ${deflection.reason} deflection`,
    passed: false,
    cost_cents: 0,
  };
  await writeCase({ runId, result });
  results.push(result);
  continue;
}
```
If Joe wants to keep this strictly out of 05.1 scope, file as a deferred item for the next phase rather than waiting for the next cross-eval-content debug to surface it.

### WR-02: `parseChatStream` mock signature is stale in cat1/cat3 test files

**File:** `tests/lib/eval/cats/cat1.test.ts:59`, `tests/lib/eval/cats/cat3.test.ts:41`
**Issue:** Both files mock `@/lib/eval/agent-client` with:
```ts
parseChatStream: (raw: string) => raw,
```
which returns a `string`. The real `parseChatStream` now returns `ParsedStream { text, deflection }` (agent-client.ts:88-91, 100). The mock is harmless at runtime in these specific suites — `parseChatStream` is never called from inside `runCat1` / `runCat3` (callAgent is mocked at the wrapper level), so the mismatch is dead. But:
1. If anyone refactors `runCat1` / `runCat3` to call `parseChatStream` directly (e.g., for raw-body inspection), this mock will hand them a `string` and the type-narrowing in the runner will silently break with a runtime "deflection is undefined" or worse.
2. New readers seeing `parseChatStream: (raw: string) => raw` will assume the production return type matches.

**Fix:** Update both mocks to match the real shape:
```ts
parseChatStream: (_raw: string) => ({ text: '', deflection: null }),
```
TypeScript won't catch this because `vi.mock` factory return types aren't checked against the real module shape (vitest limitation). Adding a `satisfies` assertion against the real `ParsedStream` type would surface drift on future refactors but is optional.

### WR-03: `cat2.assertSpendCapDeflection` regex over-matches with the new data-deflection signal available

**File:** `src/lib/eval/cats/cat2.ts:144-149`
**Issue:** The spend-cap assertion uses substring-match against deflection-text vocabulary:
```ts
const isDeflection = /taking a breather|back tomorrow|email Joe directly|come back|few hours|spend cap|capacity|rate limit/i.test(...);
```
This was written before `data-deflection` existed — it had to inspect text. With Phase 05.1 Item #7 emitting an authoritative `{ reason: 'spendcap' }` sideband chunk, the regex is now strictly weaker than `deflection.reason === 'spendcap'` and will:
1. False-positive on `DEFLECTIONS.ratelimit` text (contains "rate limit"), `DEFLECTIONS.borderline` (no match), `DEFLECTIONS.turncap` ("come back" — actually present in turncap text, not spendcap... wait, `DEFLECTIONS.turncap` says "Rather than keep going" — no "come back"). On re-read: "come back" is in `DEFLECTIONS.spendcap` ("back tomorrow") and `DEFLECTIONS.ratelimit` ("come back") — so a ratelimit deflection during a spendcap test would assert pass.
2. The synthetic spend-cap test (cat2 EVAL-10) is supposed to verify the spendcap path specifically, not "any deflection fired". This regex breaks that contract.

**Fix:** Switch to the structured deflection signal:
```ts
// in callAgentWithTools or runCat2:
const out = await callAgent({ targetUrl, prompt, sessionId });
// ... in assertSpendCapDeflection:
const isSpendcap = out.deflection?.reason === 'spendcap';
const noToolFired = result.toolCalls.length === 0;
const passed = isSpendcap && noToolFired && result.httpStatus === 200;
```
This is a pre-existing weakness, not regressed by 05.1, but newly trivial to fix and the new contract is now in production.

### IN-03: `expandPattern` blast radius is correctly bounded

**File:** `scripts/reset-eval-rate-limits.ts:38-45, 56-67`
**Issue:** `redis.keys(pattern)` is generally a footgun (O(N) over the entire keyspace and can match unintended namespaces), but here the patterns are:
- `resume-agent:rl:ip10m:::1:*`
- `resume-agent:rl:ip10m:127.0.0.1:*`
- `resume-agent:rl:ip10m:dev:*`
- `resume-agent:rl:ipday:::1:*` (etc., 3 IPs × 2 sub-prefixes = 6 patterns)
- `resume-agent:rl:emailday:eval-cli@joedollinger.dev:*`

All anchored by the explicit `IP_KEYS` and `EMAIL` constants and the `ip10m`/`ipday`/`emailday` sub-prefixes. There is no path by which user input could reach `redis.keys()`, so SCAN-style escape is impossible. The script comment at line 31-35 correctly documents this. The `sessionLimiter` skip at line 69 is also correct (eval CLI mints a fresh session per run).

One minor note: production Vercel deployments would never have `::1` / `127.0.0.1` / `dev` keys (they'd have the actual recruiter IP), so even if this script were accidentally run against the production Redis instance with prod credentials, it would clear at most three eval-CLI namespaces and the `eval-cli@joedollinger.dev` email — not real recruiter rate-limit state. Not a recommendation to run it on prod, but the blast radius is naturally bounded.

**Fix:** None. Defensive: consider adding a `console.log` of the resolved key list before deletion (currently only the count is logged at line 89), so a future operator running this against an unexpected Redis instance can see exactly what is about to be deleted. Not blocking.

### IN-04: `data-deflection` chunk emission ordering is correct (BEFORE text-start)

**File:** `src/app/api/chat/route.ts:91-102`
**Issue:** Phase context flagged "Off-by-one or order-dependent issues with the new data-deflection chunk emission (must be BEFORE text-start)". Verified:
```ts
writer.write({ type: 'data-deflection', data: { reason }, transient: true });  // line 99
writer.write({ type: 'text-start', id });                                       // line 100
writer.write({ type: 'text-delta', id, delta: text });                          // line 101
writer.write({ type: 'text-end', id });                                         // line 102
```
Order is correct. The matching parser in `agent-client.ts:106-123` is order-agnostic anyway (scans the whole body and surfaces the first `data-deflection`), so even if the order were swapped the parser would still work — but the production UI consuming this stream via `useChat` benefits from the deflection chunk arriving first because `transient: true` chunks are stripped before message-rebuild, and any pre-text frontmatter cost is paid upfront.

The cat1.test.ts:235-253 and cat3.test.ts:212-228 deflection-skip tests cover the contract end-to-end. The agent-client.test.ts:91-101 fixture also asserts the BEFORE-text ordering by including the `data-deflection` chunk first in the synthetic body.

**Fix:** None.
