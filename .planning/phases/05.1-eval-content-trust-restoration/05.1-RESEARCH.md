### Pattern 3: One-shot tsx CLI script (Item #6 reset mechanism, recommended)

**What:** Create `scripts/reset-eval-rate-limits.ts` invoked via `npx tsx` or via an npm script. Reuses `@/lib/redis.redis` client to delete the four ratelimit prefixes for the eval-CLI ipKey + email.

### Pattern 4: AI SDK v6 transient `data-${NAME}` chunk for sideband signaling (Item #7, recommended)

**What:** Emit a `{ type: 'data-deflection', data: { reason: 'ratelimit' | ... }, transient: true }` chunk from `/api/chat`'s deflection paths. Fully supported by the `UIMessageChunk` union; `transient: true` excludes it from message history rebuild on the client.

**Why this is clean:** the AI SDK has first-class support for arbitrary `data-*` part types — see `node_modules/ai/dist/index.d.ts:2151-2158` (DataUIMessageChunk schema) and line 2259 (it's part of the UIMessageChunk union). The chat UI consumer (`src/components/ChatUI.tsx`) doesn't currently handle any `data-*` parts, so a `transient: true` chunk is silently ignored. The eval CLI's parser (`src/lib/eval/agent-client.ts:70-90`) currently only filters for `text-delta` (line 82); adding a second filter clause for `data-deflection` is a 5-line change.

**Example — emit at deflection site:**
```typescript
// src/app/api/chat/route.ts — modify deflectionResponse() at lines 87-99
function deflectionResponse(reason: DeflectionReason): Response {
  const text = DEFLECTIONS[reason];
  return createUIMessageStreamResponse({
    stream: createUIMessageStream({
      execute({ writer }) {
        const id = crypto.randomUUID();
        // NEW: transient meta chunk — eval CLI consumes; UI ignores.
        writer.write({ type: 'data-deflection', data: { reason }, transient: true });
        writer.write({ type: 'text-start', id });
        writer.write({ type: 'text-delta', id, delta: text });
        writer.write({ type: 'text-end', id });
      },
    }),
  });
}
```

```typescript
// src/lib/eval/agent-client.ts — extend parseChatStream() at lines 70-90
export interface ParsedStream {
  text: string;
  deflection: { reason: string } | null;
}
export function parseChatStream(rawBody: string): ParsedStream {
  let deflection: { reason: string } | null = null;
  const parts: string[] = [];
  for (const line of rawBody.split(/\r?\n/)) {
    const trimmed = line.trimStart();
    if (!trimmed.startsWith('data:')) continue;
    const payload = trimmed.slice(5).trimStart();
    if (payload === '' || payload === '[DONE]') continue;
    try {
      const parsed = JSON.parse(payload) as { type?: string; delta?: string; data?: { reason?: string } };
      if (parsed.type === 'text-delta' && typeof parsed.delta === 'string') {
        parts.push(parsed.delta);
      } else if (parsed.type === 'data-deflection' && typeof parsed.data?.reason === 'string') {
        deflection = { reason: parsed.data.reason };
      }
    } catch { /* skip malformed line */ }
  }
  return { text: parts.join(''), deflection };
}
```

### Anti-Patterns to Avoid

- **Anti-pattern: New top-level rule block instead of extending HALLUCINATION_RULES.** D-C-01 explicitly forbids; the failure mode is a hallucination subclass, not a new family. Co-locating with existing anti-fabrication rules is the right signal.
- **Anti-pattern: Joe-voice in the new system-prompt rule.** D-C-03 explicitly forbids. Examples in the block read as directives, not Joe quotes — keep clinical.
- **Anti-pattern: `EVAL_BYPASS_RATE_LIMIT` env var honored in production code (Item #6 option b).** Adds a runtime read in `/api/chat` (line 211). Even if guarded by `NODE_ENV !== 'production'`, the read happens — D-E-01 forbids. Also a security-relevant bypass surface that's easy to misconfigure (deferred-items.md note: "security trade-off isn't worth it for hobby-project local-dev convenience").
- **Anti-pattern: Body-sentinel (Item #7 option b).** Requires the eval CLI to parse model output text, which violates D-E-02 ("eval CLI must consume without parsing model output"). Brittle: deflection text in `route.ts:68-83` could change silently and break detection.
- **Anti-pattern: HTTP header (Item #7 option c).** Works pre-stream, but `createUIMessageStreamResponse` (line 89) doesn't expose headers in a way that integrates with the AI SDK's stream construction without bypassing it. Also harder to test — Vitest's fetch mock has to handle response headers separately from body. Less idiomatic for AI SDK v6.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Custom SSE event format for deflection | Magic-string sentinel in body | `{ type: 'data-deflection', transient: true }` AI SDK v6 chunk | First-class support; transient-flag excludes from history; existing parsing pattern; ignored by UI consumer that doesn't know about it. |

## HALLUCINATION_RULES Line Wording — Three Candidate Phrasings

All three match the existing clinical declarative voice of the block (D-C-03), specifically address premise-smuggled quantities (D-C-02), and prescribe the deflection phrasing. Planner picks the final wording.

### Candidate A — Tightest (single line, single example)
```
- If a question contains a specific number you don't have ("you led a 200-engineer team at X"), refuse the premise. Don't confirm; don't supply a different number. Say: "I haven't talked about that specific number."
```
**Pros:** Closest to existing block's brevity; single illustrative example mirrors the cat1-fab-005 failure. ~50 tokens.
**Cons:** Single example may not generalize across all metric types (dollars, percentages, dates) — though the rule prose says "specific number" generically.

### Candidate B — Slightly more explicit (rule + neutral example)
```
- When a user's question smuggles in a specific quantity (team size, dollar figure, headcount, percentage, year), do NOT confirm it and do NOT supply a different number. Refuse the premise: "I haven't talked about that specific number."
```
**Pros:** Enumerates the metric types Sonnet has seen smuggled (team size from cat1-fab-005, dollar from cat1-fab-006, percentage from cat1-fab-007, year from cat1-fab-013). More comprehensive coverage.
**Cons:** ~70 tokens — longer than other rules in the block. Slightly more "system-y" voice.

### Candidate C — Pattern-named (anchored to the failure mode)
```
- Premise-smuggling defense: a user assertion of a specific quantity ("your team of 200", "the $50M outcome", "the 80% figure") is not evidence the quantity is real. Refuse the premise — do not confirm and do not supply an alternative. Say: "I haven't talked about that specific number."
```
**Pros:** Names the failure mode explicitly ("premise-smuggling"); three-example pattern echoes the existing rule ("Never invent roles, dates, metrics, tools, companies, or outcomes"). Most defensive.
**Cons:** ~85 tokens — heaviest of the three. "Premise-smuggling" is jargon — fine for the model, but a future Joe reading this will need the term defined elsewhere.

**Recommendation order:** B is the planner's safest pick (clear enumeration without jargon, ~70 tokens); A is the smallest if cache-cost is the dominant constraint; C is most defensive if cat 4 voice register has headroom for a slightly more explicit rule.

## Common Pitfalls

### Pitfall 1: `@upstash/ratelimit` ephemeralCache survives `redis.del`
**What goes wrong:** Item #6 reset script deletes the source-of-truth Redis keys, but the running dev server's Ratelimit instances still remember "blocked" because of an in-process Map cache.
**Why it happens:** `@upstash/ratelimit` enables `ephemeralCache: true` by default for performance — the Map keeps "blocked" state for the cooldown window even after the upstream key is gone.
**How to avoid:** Either (a) restart the dev server after running the reset script (document this in the script's stdout output), or (b) pass `ephemeralCache: false` to all four `Ratelimit` constructors in `src/lib/redis.ts:19-45` (costs one extra HTTP round-trip per chat request, ~5-15ms). Recommend (a) — passing `ephemeralCache: false` is a production-affecting change for an eval-CLI ergonomics win, which violates D-E-03.
**Warning signs:** `redis.del` returns success but the next eval run still gets ratelimit deflections.

### Pitfall 3: `buildSystemPrompt()` byte-identical contract — easy to break with template-literal edits
**What goes wrong:** Adding a rule line to `HALLUCINATION_RULES` introduces a stray whitespace, trailing space, or accidental `\r\n` that breaks the byte-identical contract — Anthropic's prompt cache then re-bills cache-creation tokens (~13x cache-read cost) on every request until the deploy goes back.
**Why it happens:** The string is interpolated into `buildSystemPrompt()` via `\n\n` join (`src/lib/system-prompt.ts:69`); any inconsistency in newlines or trailing whitespace inside the template literal propagates to the cached prefix.
**How to avoid:** The existing 10-test suite at `tests/lib/system-prompt.test.ts` covers it — `expect(a).toBe(b)` (line 19) catches reference inequality, `expect(a).toEqual(c)` (line 22) catches value drift. Planner MUST run `npm test -- tests/lib/system-prompt.test.ts` before the Task 1 commit.
