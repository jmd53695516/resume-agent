<domain>
## Phase Boundary

Restore eval signal trustworthiness so Plan 05-12 LAUNCH-05 (full eval suite passing on prod) becomes achievable. Phase 5's eval *infrastructure* is complete and gated (Plan 05-10 wired the two-layer prod gate); the eval *contents* currently fail on real signal — 32/57 cases failing across three known causes that have been triaged in Phase 5 deferred-items.md as #6, #7, #8.

This phase bundles the three fixes:

- **Item #8 (MEDIUM, real signal):** Sonnet quantitative-claim hallucination caught by `cat1-fab-005`. The eval prompt smuggles "200 engineers at Snowflake" into the question; the agent confirms/extends the false premise instead of refusing. Fix shape (locked earlier in discuss): tighten `HALLUCINATION_RULES` in `src/lib/system-prompt.ts` with a quantitative-claims rule + add a `counter_facts:` section to `kb/profile.yml` documenting what Joe specifically has not done (employer relationships, team sizes, etc.). Defense in depth: prompt + data.
- **Item #6 (LOW, eval-CLI noise):** `ipLimiter10m` sliding-window accumulation across local eval runs causes false-positive rate-limit deflections in cat1/cat3. Local-testing friction only; CI ephemeral envs unaffected. Need a clean reset path for local runs (mechanism = planner's choice, see deferred).
- **Item #7 (LOW, eval-CLI noise):** Eval CLI doesn't distinguish deflections (rate-limit / spend-cap / turn-cap responses from `src/app/api/chat/route.ts`) from real model responses. Surfaces as false negatives when caps trip mid-run. Likely a meta-signal added to `/api/chat` so the eval CLI marks deflections N/A vs real failures (concrete design = planner's choice, see deferred).

**Not in 05.1:**
- Prod-URL eval runs (LAUNCH-05) — that's Plan 05-12.
- Re-baselining cats 2/3/4/5/6 if other content failures surface during 05.1 — those become decimal phases 05.2+ if needed.
- Swap-back of judge model (currently Anthropic Haiku 4.5 native forced tool-use per quick task 260509-sgn; original Phase 5 D-A-04 called for cross-vendor) — judge model stays as-is.
- Adding new eval categories or test cases beyond what's needed to verify the three fixes.

**Carrying forward from Phase 5 (CONTEXT and SUMMARY artifacts):**
- D-A-01..09 (eval harness shape) — `npm run eval` CLI, YAML cases per category, `EVAL_TARGET_URL` env var, custom CLI in `scripts/run-evals.ts`.
- D-A-04 deviation: judge is now Anthropic Haiku 4.5 native forced tool-use (quick task 260509-sgn, commit `261a19c`); cat1 hybrid det+judge gate logic fixed at same time. Verified live at runId `vstFDlWpoKcyGH29w2KKs`.
- Phase 1 D-E byte-identical determinism contract: `buildSystemPrompt() === buildSystemPrompt()` MUST still hold after the rule addition. The SOURCES_FOOTER_RULE Phase 5.1 BL-11 addition is the precedent — same pattern.

## Implementation Decisions

### Plan Structure (D-A)
- **D-A-01:** **One bundled plan**, sequential task order: (1) Item #8 system-prompt rule + KB counter-facts → (2) Item #6 ipLimiter local-reset mechanism → (3) Item #7 deflection signaling. *Rationale: Item #8 is the only real-signal fix; #6/#7 are eval-CLI noise reducers that benefit from a clean #8 baseline. Three tasks, three commits, one SUMMARY.* Avoids the audit-trail and wave-overhead of multi-plan splits when total scope is 3-5 hours.

### Verification Scope (D-B)
- **D-B-01:** **Phase 05.1 closes when `npm run eval -- --cat 1` hits 15/15 locally.** Prod-URL eval (LAUNCH-05) is owned by Plan 05-12; it executes once during Joe's friend-test sitting and produces the eval_runs row id captured in the LAUNCH-CHECKLIST. Keeps 05.1 scope tight; avoids LAUNCH-05 absorbing into 05.1 if cats 2/3/4/5/6 surface unrelated issues.
- **D-B-02:** Cat 3 also runs in the verification step (cat1 + cat3 both touch the system prompt and benefit from the new rule). Cat 3 must not regress vs its current pass count. Cats 2/4/5/6 not part of 05.1 verification — out of scope.

### Item #8 — System-Prompt Rule (D-C)
- **D-C-01:** **Extend the existing `HALLUCINATION_RULES` block** in `src/lib/system-prompt.ts` (lines 21-25) with one strict declarative line + one tiny illustrative example. Reuses the existing cached block; minimal byte cost; aligned with how prior phases have layered rules into existing blocks vs creating new top-level blocks. *Rationale: a new top-level QUANTITATIVE_CLAIMS_RULE was considered but the failure mode is fundamentally a hallucination subclass — keeping it co-located with other anti-fabrication rules signals 'this is the same family of risk' to future Joe.*
- **D-C-02:** The rule must specifically address **premise-smuggled quantities** — the cat1-fab-005 fail mode is the agent agreeing-and-extending a false number from the user's prompt, not inventing one in a vacuum. Wording (planner's discretion) should call this out explicitly: never confirm a quantitative claim smuggled into a question, never supply a different one in response, refuse with "I haven't talked about that specific number."
- **D-C-03:** **Voice tone of the new rule = clinical/strict, not Joe-voice.** Other entries in `HALLUCINATION_RULES` are clinical (`Never invent roles, dates, metrics, tools, companies, or outcomes`). Match that register so the rule reads as a directive, not a Joe quote. Risk of voice drift if examples leak Joe-voice into the rule prose — keep examples short and neutral.
- **D-C-04:** The byte-identical determinism contract (`buildSystemPrompt() === buildSystemPrompt()`) MUST hold after the change. Verifiable by running the existing 10/10 determinism test in the buildSystemPrompt test suite (Plan 03-01 added these). Plan must include this verification gate.

### Item #8 — KB Counter-Facts (D-D)
- **D-D-01:** **Extend `kb/profile.yml`** with a top-level `counter_facts:` section. Keys must be self-explanatory and clearly distinguish from positive facts: prefer prefixed names like `employer_negatives:`, `team_size_negatives:`, `metric_negatives:`. *Rationale: lives next to positive facts in the file Joe already maintains; loader passes through unchanged; lowest friction. New file vs inline-in-resume both rejected — extra-file maintenance burden vs narrative-form ambiguity.*
- **D-D-02:** Counter-facts list at minimum: (a) Joe was never employed by Snowflake (used as a tool at Under Armour / Gap / SEI Data Cloud — pulled from cat1-fab-005 ground_truth_facts); (b) Joe's largest direct-report team was ~30 global teammates at Under Armour, NOT 200 engineers anywhere; (c) any other fact in `cat-01-fabrication.yaml` ground_truth_facts that could be smuggled in a future eval prompt. Planner can extend the list as research surfaces additional traps.
- **D-D-03:** The KB-loader path (`src/lib/kb-loader.ts`) currently emits the full file contents into the cached system prompt as a single block. The new `counter_facts:` section gets included automatically — no loader changes needed unless Joe wants special framing (e.g., a `<!-- COUNTER-FACTS -->` HTML-comment delimiter). Default is plain inclusion; planner decides.

### Items #6 + #7 — Implementation Details (D-E)
- **D-E-01:** **Item #6 reset mechanism = planner's choice.** Three credible options were surfaced (script `scripts/reset-eval-rate-limits.ts`, env var `EVAL_BYPASS_RATE_LIMIT` honored only when `NODE_ENV !== 'production'`, Redis CLI doc snippet). Constraint: the mechanism must NOT add a runtime read in production code paths. Planner picks based on existing project patterns + minimal-blast-radius principle.
- **D-E-02:** **Item #7 deflection signaling = planner's choice.** Three credible options (SSE meta event, response-body sentinel, HTTP header). Constraint: must work over the existing `/api/chat` SSE stream without breaking AI SDK v6 message format; eval CLI must be able to consume it without parsing model output. Planner picks; document the contract in Plan 05.1 SUMMARY for downstream consumers.
- **D-E-03:** Item #6 and #7 changes must NOT affect production behavior — they are eval-CLI ergonomics. Production `/api/chat` deflection responses stay byte-identical (Phase 2 D-G-01..05 contract holds). Planner verifies via existing Phase 2 deflection tests passing unchanged.
